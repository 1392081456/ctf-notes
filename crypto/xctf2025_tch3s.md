# 9th XCTF International Finals — Tch3s (English)

Custom 11-round 128-bit block cipher. The key is generated by `srand(time(NULL))`; the output file ships an encrypted flag plus 100,000 plaintext/ciphertext/timing tuples. The challenge name hints at a **timing side channel** — that's a decoy. The real weakness is the **predictable PRNG seed**.

Concepts: [[concepts/predictable-srand-time-seed]], [[concepts/gdb-call-binary-decrypt]]

## Files

| File | Size | Contents |
|---|---|---|
| `Tch3s` | 24 KB | stripped PIE x86_64 ELF |
| `output` | 21 MB | encrypted flag + 100,000 test triples (plain/cipher/timing) |

Binary flow:
1. `srand(time(NULL))` → 16 bytes of key via `rand() & 0xff`
2. `fcn.27b4(expanded_keys, key)` runs the key schedule (208 B expanded)
3. `getenv("FLAG")` reads the real flag (default fallback otherwise)
4. `fcn.30ba(IN, OUT, expanded_keys)` encrypts the flag in 16-byte ECB blocks
5. Prints `encrypted flag: …`
6. Loops 100,000 times: random plaintext → encrypt + decrypt + timing print

## Key observation

> **Timing is the decoy**: the name screams "time-channel attack", but `srand(time(NULL))` uses the program-start Unix timestamp — directly **brute-forcible**.

## Steps

### 1. Brute-force the seed

`srand(seed)` + 16 × `rand() & 0xff` = key. Next 16 = Test 1 plaintext (`720B4455C91B6A024135343386B6679D`, known from the output file). Scan all of 2024–2026 (~95 M seconds, ~1 minute):

```python
import ctypes
libc = ctypes.CDLL('libc.so.6')
target_pt = bytes.fromhex('720B4455C91B6A024135343386B6679D')

for seed in range(int(time.mktime((2024,1,1,0,0,0,0,0,0))),
                  int(time.mktime((2026,12,31,23,59,59,0,0,0)))):
    libc.srand(seed)
    for _ in range(16): libc.rand()      # consume key bytes
    pt = bytes(libc.rand() & 0xff for _ in range(16))
    if pt == target_pt:
        print(f"seed = {seed}")
        break
```

**Found**: `seed = 1753843495` (2025-07-30 02:44:55 UTC).

### 2. Recover the key

```python
libc.srand(1753843495)
key = bytes(libc.rand() & 0xff for _ in range(16))
# 17a7c56d2dde47fdbdceb68060b9169c
```

### 3. Call the binary's own decrypt function

Too lazy to reimplement the cipher. Inject our recovered key into the running process via gdb-python, then `call` the decrypt function directly:

```python
gdb.execute('starti')
base = <PIE base>
KEY = bytes.fromhex('17a7c56d2dde47fdbdceb68060b9169c')
CT  = bytes.fromhex('B789EB607A91D08888E2C4C4E4D9573FF5333E4783390B91EDB9D2B4FD2A5FC0')

# Break right before the call to fcn.27b4, overwrite the 16-B random key
gdb.Breakpoint(f'*{base+0x166b:#x}')
gdb.execute('continue')
key_buf  = $rsi
exp_keys = $rdi
inferior.write_memory(key_buf, KEY)

# Let fcn.27b4 run, computing the real expanded_keys from our key
gdb.Breakpoint(f'*{base+0x1670:#x}')
gdb.execute('continue')

# Save expanded_keys (decrypt mutates it)
exp_keys_saved = inferior.read_memory(exp_keys, 0xd0)

# Write ciphertext, call decrypt twice (one per block)
in_addr  = malloc(64); write(CT)
out_addr = malloc(64)
dec_addr = base + 0x31a1

# fcn.31a1(IN, OUT, KEYS) — same arg order as encrypt
gdb.execute(f'call (long(*)(void*,void*,void*))(...)({in_addr},   {out_addr},   {exp_keys})')
inferior.write_memory(exp_keys, exp_keys_saved)        # restore before block 2
gdb.execute(f'call (long(*)(void*,void*,void*))(...)({in_addr+16},{out_addr+16},{exp_keys})')

print(inferior.read_memory(out_addr, 32))
```

## Pitfalls

- **`fcn.30ba` and `fcn.31a1` share signature `(IN, OUT, KEYS)`** — not the OpenSSL-style `(OUT, IN, KEY)`. The encrypt site is the clearest tell: `rdi = plain_in`, `rsi = cipher_out`; the decrypt site mirrors that. To disambiguate, look at fcn.30ba's tail: `mov [rcx], rax; mov [rcx+8], rdx` where `rcx` = arg2 — arg2 is the OUTPUT.
- **Restore `expanded_keys` between decrypt calls.** `fcn.31a1` mutates the key buffer (typical AES-style temporary inverse-round-key reuse). Without restoring, block 2 decrypts to garbage.
- **Get the PIE base from `info proc mappings`'s first segment**; every breakpoint must be `base + offset`.
- **The 100 K timing data is a big red herring.** The author wants you to chase a cache-timing side channel; in practice the PRNG is so weak that a 95 M-second brute force solves it in under a minute.
- **glibc `rand()` differs across versions.** Tested on GLIBC 2.34 on Kali, matching the challenge libc. Across distros, verify with a known seed first (`srand(123)` then 16 × `rand()&0xff`).

## Full story

1. Author uses `srand(time(NULL))` to seed the cipher key, brags about timing with 100 K test cases
2. Solver realizes `time(NULL)` is brute-forcible at second granularity
3. Use Test 1 plaintext to fingerprint the seed
4. Re-derive the key from the seed
5. Inject the key into the live binary and call its own decrypt

Not a single line of timing analysis was needed, yet the flag literally says `timing_attack_is_dangerous` — a self-troll from the author.
